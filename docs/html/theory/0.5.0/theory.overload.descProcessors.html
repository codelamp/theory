theory.overload .descProcessors

Overload description processors allow the overload behaviour to be extended.

These should make changes to the description object before being handed to the item processors.

You can add your own processor by .push()-ing a method to this array. The method should have the following signature:

function(desc, internal) {}
  • desc is the entire description object
  • internal is a provided storage object, per description but separate from description

It should be noted that the order of processors can change expected outcomes. Shift the orders only if you know what you are doing.

theory.overload.descProcessors.inlineNames()

Inline names allows for a more succint way of naming params.

So rather than:

[array, callable, number] > [list, callback, delay]

You can use:

[array:list, callable:callback, number:delay]

It is personal preference which is prefered for readability. However, inline names are less prone to typing mistakes. Especially if you start using optional parameters, for example:

[array][callable, number]? > [list][callback, delay]?

Would be:

 [array:list][callable:callback, number:delay]?

theory.overload.descProcessors.translateToArgsObject()

translateToArgsObject allows methods defined to receive an args object with named attributes, instead of a list of params.

For example:

var test = t.overload({
  '[array, callable] > [list, callback]': function(args){
    console.log(args)
  };
});

test([123], function(){});

Would produce the following in the console:

{ list: [123], callback: function(){} }

I personally prefer the readability of the above to that of the colon syntax with Inline Names. However, it can get a bit verbose.
